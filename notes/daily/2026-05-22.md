## Core Takeaways

- [ ] **Agent = constrained execution loop, not autonomy.** Goals + tools + state + permissions + stop conditions. All five required. Model + tools without boundaries = a talking script bomb.
- [ ] **Tools are more dangerous than answers.** Wrong text = wrong answer. Wrong tool call = wrong action. Read-only tools vs write-to-chain tools are not the same risk level.
- [ ] **Most dangerous design**: vague goals × broad tools × persistent memory × large asset permissions. AI×Web3 Agents must split: auto-execute read steps, human-check write steps.

## Notes & Takeaways

### Agent：核心不是让模型更像人，而是让执行循环有清楚边界。  
    建立分工：模型负责提出候选行动，系统负责限制行动空间，用户负责批准高风险边界。
- 第一性原理：由于agent是操作而不是回答，所以更要小心，设立边界告诉Agent能做什么、不能做什么、做完如何验证、失败如何停止、谁能审计它做过的事。
  - 工具比回答更危险：可操作的风险要远大于读取、检索
  - 状态必须外置和可查：每一步节点，执行状态都要有记录
  - 停止条件要明确：达到目标、超出预算、信息不足、风险越界、用户拒绝，都应该让 Agent 停下来。
- tool use：agent可以调用外部工具：搜索、数据库、API、代码执行、邮件、支付接口、内部系统等，让agent能做事。
  - 工具设计要明确：
    - 输入 schema
    - 权限范围
    - 是否只读
    - 是否会产生外部副作用
    - 调用前后如何记录
    - 哪些调用需要人工确认
- planning：将需要agent做的事列成计划，更详细更有流程的指挥agent怎么做。
  - 一个好的 Agent plan 应该能暴露：
    - 每一步需要什么工具
    - 每一步读还是写
    - 哪些步骤可自动执行
    - 哪些步骤需要用户确认
    - 失败时是否可以重试
    - 最终如何验证任务完成
- state：agent当前执行任务的状态和情况，用state看户目标、已完成步骤、工具返回、错误、预算、确认记录和最终输出。需要可查询、可恢复、可审计。
- reflection：模型自检，查看是否有错误（信息不足、工具失败、计划不合理等）。用于提高质量，但不能替代外部验证。
- multi-agent：一个任务交给多个agent分工。适合复杂任务，但容易造成上下文传递丢失、责任边界模糊、一个 Agent 的错误被另一个 Agent 当成事实、工具权限扩散等问题。因此要先确定multi-agent能减少复杂度再使用。
- 在ai web3中的位置：读取用户目标生成计划，并执行。（再确认什么可以执行什么是高风险操作需要用户确认的前提下。）
